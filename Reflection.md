# Reflection

The main challenge in building this Bank application was designing the class to correctly encapsulate internal state while keeping methods clean and predictable. Deciding how to structure the `accounts` getter — whether to expose a live reference or a defensive copy — seemed like a minor detail at first, but it turned out to be the most consequential design decision in the entire project. Balancing input validation across `createAccount`, `deposit`, and `withdraw` also required careful thought to ensure edge cases like zero amounts, negative values, and empty names were all handled consistently without duplicating logic.

Debugging the failing tests (TP-005, TP-008, TP-010, TP-015) was particularly tricky because the `createAccount` method appeared to work correctly — it returned a well-formed account object every time. The bug only surfaced when a second operation was performed on that account, which made the root cause non-obvious. The issue was that the `get accounts()` getter returned `[...this.data.accounts]`, a shallow copy, so `push()` calls inside `createAccount` were adding accounts to a temporary copy rather than the real internal array. No account was ever actually stored, but nothing failed until you tried to look one up.

This project reinforced how easy it is to introduce subtle mutation bugs in JavaScript when spread syntax is used carelessly inside a class. The key lesson is that defensive copying belongs at the outward-facing boundary — to protect internal state from external callers — not inside internal methods that need to mutate that same state. It also highlighted the value of writing tests that chain multiple operations together, since isolated unit tests on `createAccount` all passed and gave a false sense of correctness; the bug only became visible through tests that followed creation with a deposit, withdrawal, or balance check.
